# V4.0 真实业务验收与生产可靠性设计

> 日期：2026-06-22  
> 状态：已批准，待实施计划  
> 基线：`origin/main@91e906c`  
> 产品定位：可信、自动、可运营的企业财税执行平台

## 1. 目标与边界

V4.0 不再以增加页面或堆叠新模块为主，而是验证并提升现有系统的正式可用性。

目标部署形态：

- 单集团私有云部署。
- 多部门或分子公司。
- 30–100 名用户。
- 部分银行、发票、税务接口使用真实凭证，其余使用沙箱或文件交换。
- 规则满足时大部分流程可自动执行。
- 税务正式申报、对公付款、工资实际发放、凭证正式过账及反结账、红字发票、收入调整、银行指令提交等高风险动作保留授权人最终确认。
- 保留未来升级为多企业 SaaS 所需的租户隔离、计费和配额接口。

明确不在首期承诺：

- 无企业证书时替代官方税务系统完成最终法定申报。
- 无银行签约时执行真实资金划拨。
- 未经人工授权自动完成高风险财税动作。
- 与真实业务验收无关的新业务模块扩张。

## 2. 当前系统基线

最新主线已完成：

- V2 财税业务主链。
- V3 全页面结构化、响应式和无障碍升级。
- 银行、发票、税务文件交换与外部连接器框架。
- 银行对账、工资代发与社保联动。
- 月度结账、全局期间、统一待办、全局搜索和导航角标。
- AI 会计、资料完整性和审计勾稽 Agent。
- 现金流预测、往来单位画像和申报到期提醒。
- 健康检查、性能索引、订阅计费和反馈模块。
- 全站财税业务链路条和财税资料包。

V4 必须复用这些能力。只有真实验收证明现有实现不足时，才允许修改领域模型或重构模块。

当前需要优先治理的工程事实：

- `apps/api/src/app.ts` 体量过大，路由调度应按领域拆分。
- `apps/web/src/lib/api.ts` 体量过大，客户端应按领域拆分。
- 浏览器业务验收不足，代码测试不能证明用户流程可用。
- 异步处理仍缺统一运行时和状态模型。
- 文件服务、密钥管理、身份安全、恢复演练尚未达到正式私有云要求。
- AI 输出已有留痕，但缺少正式评测集、质量门禁和成本治理。

## 3. 设计原则

### 3.1 纵向切片交付

每次完整打通一条真实业务，而不是分别开发页面、后端、单据或测试。

每条切片同时交付：

- 用户入口与操作引导。
- 业务状态机与审批。
- 表格、详情和跨页穿透。
- 原始资料、正式单据和附件。
- 凭证、总账、报表和税务处理。
- 银行、发票或其他外部连接。
- 异常处理、风险和审计证据。
- 浏览器、API、数据库、权限、性能和恢复测试。

### 3.2 自动化边界

系统可自动完成：

- 资料识别和分类。
- 资料完整性检查。
- 重复发票和金额差异检查。
- 任务生成、催办和 SLA 升级。
- 会计科目、税务处理和凭证草稿建议。
- 低风险候选匹配。
- 资料包生成和归档准备。
- 外部连接器的沙箱调用、文件生成和失败重试。

必须人工最终授权：

- 税务正式申报。
- 对公付款和工资实际发放。
- 凭证正式过账、反结账和重大调整。
- 红字发票和收入调整。
- 银行指令正式提交。

### 3.3 用户界面标准

每个业务页面必须包含：

1. 顶部业务链路条：显示上一步、当前环节、下一步和总体完成度。
2. 当前状态摘要：责任人、期限、金额、缺失资料、阻塞原因。
3. 一个明确的主操作按钮。
4. 可筛选和可排序的业务表格。
5. 右侧详情抽屉，集中展示资料、审批、单据、凭证、税务、风险和审计时间线。
6. 空态、加载态、错误态和只读态。
7. 所有状态使用统一中文标签，不暴露内部枚举。
8. 所有跨页导航保留业务上下文，并可返回原对象。

错误不能只显示失败消息，必须同时显示：

- 用户可读原因。
- 稳定的技术错误码。
- 受影响对象。
- 责任角色。
- 推荐修复动作。
- 是否可以重试。
- 重试入口或补件入口。

## 5. 切片一：员工购买物品报销

### 5.1 业务流程

`员工提交 → AI 识别 → 主管审批 → 财务复核 → 资产/费用判断 → 凭证 → 付款或冲借款 → 税务留档`

### 5.2 角色

- 员工或经办人。
- 部门负责人。
- 采购或行政。
- 会计。
- 出纳。
- 财务负责人或最终授权人。

### 5.3 输入资料

- 费用报销申请。
- 发票。
- 支付凭据。
- 物品清单。
- 用途说明。
- 采购审批或临时采购说明。
- 涉及固定资产时的验收、领用和使用部门信息。

### 5.4 系统单据

- 费用报销单。
- 报销票据包。
- 采购记录。
- 验收记录。
- 付款申请单。
- 资产卡片候选。
- 进项税处理记录。

### 5.5 会计与税务处理

系统必须判断：

- 办公费用。
- 低值易耗品。
- 存货。
- 固定资产。
- 福利或非经营性支出。
- 进项税可抵扣、不可抵扣或待确认。

输出：

- 会计处理建议及依据。
- 凭证草稿。
- 付款或其他应付款处理。
- 企业所得税税前扣除提示。
- 增值税进项处理。

### 5.6 表格要求

报销台账至少显示：

- 报销编号。
- 经办人和部门。
- 发生日期。
- 费用类型。
- 金额和税额。
- 发票状态。
- 资料完整度。
- 审批状态。
- 会计处理状态。
- 付款状态。
- 当前责任人。
- 下一步。

### 5.7 异常用例

- 发票抬头错误。
- 无票或票据不完整。
- 同一发票重复提交。
- 金额不一致。
- 用途不清。
- 超审批标准。
- 经营无关支出。
- 应资本化但被费用化。
- 审批人与付款人为同一人且违反职责分离。

## 6. 切片二：员工差旅报销

### 6.1 业务流程

`出差申请 → 行程发生 → 票据归集 → 费用拆分 → 标准校验 → 审批 → 凭证和付款 → 税务留档`

### 6.2 角色

- 出差人。
- 报销经办人。
- 部门负责人。
- 行政或人事。
- 会计。
- 出纳。
- 财务负责人。

### 6.3 输入资料

- 出差申请和审批。
- 机票、火车票或其他交通票据。
- 行程单。
- 酒店发票和住宿明细。
- 餐饮发票。
- 市内交通凭据。
- 出差人员、地点和事由。
- 多人出差时的人员清单和费用分摊。

### 6.4 系统单据

- 差旅报销单。
- 差旅票据包。
- 差旅费用明细。
- 超标审批单。
- 业务招待拆分单。
- 补贴和个人承担明细。

### 6.5 会计与税务处理

系统必须区分：

- 长途交通。
- 住宿。
- 市内交通。
- 差旅补贴。
- 出差餐费。
- 业务招待。
- 会议或培训。
- 个人消费。

输出：

- 费用分类结果。
- 税前扣除判断。
- 进项抵扣判断。
- 员工借款或垫付款冲销。

### 6.6 表格要求

差旅台账至少显示：

- 出差单号。
- 出差人员。
- 部门。
- 起止日期和地点。
- 事由。
- 交通、住宿、餐饮、补贴和其他金额。
- 标准内金额和超标金额。
- 资料完整度。
- 审批、凭证和付款状态。
- 当前责任人和下一步。

### 6.7 异常用例

- 无出差审批。
- 发票日期和行程不一致。
- 地点不一致。
- 超标准住宿或交通。
- 多人费用混报。
- 业务招待混入差旅。
- 重复票据。
- 个人消费混入。
- 行程变更但未补说明。

### 7.1 业务流程

`合同审批 → 履约或验收 → 开票申请 → 发票开具 → 回款对账 → 收入确认 → 税务申报 → 应收风险`

### 7.2 角色

- 业务人员。
- 业务负责人。
- 法务。
- 财务。
- 开票岗。
- 出纳。
- 税务会计。
- 最终授权人。

### 7.3 输入资料

- 合同和补充协议。
- 客户主数据及开票信息。
- 履约、交付或验收依据。
- 开票申请。
- 收款计划。
- 银行回单。
- 收入确认依据。

### 7.4 系统单据

- 合同审批单。
- 履约或验收记录。
- 开票申请单。
- 销项发票。
- 收款认领单。
- 收入确认单。
- 应收催收记录。
- 税务申报资料。

### 7.5 会计与税务处理

系统必须处理：

- 应收账款。
- 合同负债或预收款。
- 主营业务收入。
- 销项税。
- 先开票、先收款和先履约的时点差异。
- 部分履约和分期开票。
- 银行流水自动匹配。
- 逾期应收风险。

### 7.6 表格要求

合同收入台账至少显示：

- 合同编号和客户。
- 合同金额和税率。
- 履约进度。
- 已开票、未开票。
- 已收款、未收款。
- 已确认收入、待确认收入。
- 应收账龄。
- 税务处理状态。
- 风险等级。
- 当前责任人和下一步。

### 7.7 异常用例

- 无合同或未审批合同申请开票。
- 合同、履约和发票内容不一致。
- 税率或客户信息错误。
- 先开票、先收款或先确认收入产生时点差异。
- 同一履约成果重复确认收入。
- 已开票长期未回款。
- 回款无法认领。
- 红字或作废后数据未同步。

## 8. 单据与资料标准

### 8.1 通用字段

所有正式单据必须包含：

- 企业和组织信息。
- 单据编号与版本。
- 业务事项编号。
- 申请人、经办人、责任部门。
- 业务日期和会计期间。
- 金额、税额、币种。
- 事由和用途。
- 资料清单及完整度。
- 审批记录。
- 关联凭证、税务事项和付款记录。
- 创建、修改、归档时间。
- 打印、导出和校验信息。

### 8.2 字段来源

每个字段应标记来源：

- 用户录入。
- OCR 或 AI 识别。
- 系统计算。
- 外部接口。
- 人工复核。

AI 或 OCR 字段必须可查看原文证据和置信度。

### 8.3 版本和归档

- 单据修改生成版本记录。
- 已审批或已归档版本不可静默覆盖。
- 附件保存哈希值。
- 资料包提供清单、对象 ID、文件哈希和生成时间。
- 打印版与页面数据必须来自同一结构化模型。

## 9. 工作流、审批和授权

### 9.1 状态模型

通用状态：

- `draft`
- `collecting_documents`
- `ready_for_review`
- `under_review`
- `awaiting_authorization`
- `executing`
- `completed`
- `blocked`
- `cancelled`
- `correcting`

状态变化必须记录：

- 前状态和后状态。
- 操作人或自动化主体。
- 时间。
- 依据。
- 规则版本。
- 相关资料。

### 9.2 职责分离

系统至少支持：

- 申请人与审批人分离。
- 会计与出纳分离。
- 制单与审核分离。
- 审核与过账分离。
- 指令准备与最终授权分离。

违反策略时不得只提示，应阻止高风险动作。

### 9.3 自动执行

每个自动命令必须具备：

- 幂等键。
- 业务对象版本。
- 执行主体。
- 输入快照。
- 结果快照。
- 重试策略。
- 超时策略。
- 补偿或人工接管策略。

## 10. 异步任务运行时

OCR、AI、导出、对账、资料包和外部连接器统一采用：

- `waiting`
- `running`
- `succeeded`
- `failed`
- `cancelled`

每个任务记录：

- 任务类型。
- 业务对象。
- 幂等键。
- 当前步骤和进度。
- 尝试次数。
- 下次重试时间。
- 最后错误码和错误详情。
- 输入与输出引用。
- 发起人和授权人。

用户必须能查看进度、取消允许取消的任务、重试可重试任务，并从失败任务返回修复入口。

## 11. AI 治理

### 11.1 留痕

每次 AI 运行记录：

- 模型供应商和模型。
- Prompt 模板和版本。
- 系统规则版本。
- 输入资料引用。
- 输出。
- 置信度。
- Token 和成本。
- 人工修改。
- 最终是否采纳。

### 11.2 评测

建立固定评测集：

- 采购费用分类。
- 固定资产判断。
- 差旅和招待拆分。
- 发票重复和异常。
- 合同开票时点。
- 收入确认时点。
- 税务风险。

上线门禁：

- 资料完整性识别召回率不低于 95%。
- AI 会计建议人工采纳率目标不低于 85%。
- 高风险误自动执行为 0。
- 模型或 Prompt 更新必须完成回归评测。

## 12. 外部连接器

统一连接器接口应包含：

- `healthCheck`
- `validateConfiguration`
- `pull`
- `push`
- `getStatus`
- `retry`
- `downloadExchangeFile`
- `parseExchangeFile`

运行模式：

- 真实接口。
- 沙箱。
- 文件交换。

需要覆盖：

- 银行流水拉取。
- 银行付款或工资代发指令。
- 发票验真与状态更新。
- 税务申报文件生成和状态回写。
- 社保、公积金和个税文件交换。

## 13. 测试与发布门禁

### 13.1 功能门禁

- 三条标准路径浏览器 E2E 100% 通过。
- 关键边界和异常用例通过率不低于 95%。
- 页面、导航、单据、打印和资料包完整。
- 浏览器后退、刷新和分享链接保持上下文。

### 13.2 权限门禁

- 集团、公司和部门数据隔离。
- 申请、审批、记账、付款职责分离。
- 高风险动作最终授权。
- 越权请求全部拒绝并写入审计日志。

### 13.3 数据门禁

- 数据库事务一致。
- 重复提交不产生重复凭证、付款或申报。
- 金额、税额、币种和期间精确。
- migration 可在空库和升级库执行。
- 备份数据可恢复并通过业务核对。

### 13.4 性能门禁

按 30–100 用户负载模型：

- 核心页面 P95 不超过 2 秒。
- 普通 API P95 不超过 500ms。
- 长任务异步执行并显示进度。
- 数据库慢查询有阈值和告警。

### 13.5 恢复门禁

- 异步任务可重试或续跑。
- 外部接口断线后可补传。
- RPO 不超过 24 小时。
- RTO 不超过 4 小时。
- 至少完成一次书面记录的恢复演练。

### 13.6 审计门禁

- 输入、规则、AI、人工结论全链路可追溯。
- 单据版本和审批记录不可丢失。
- 高风险授权动作 100% 有证据。
- 资料包可校验文件哈希和对象清单。

## 15. Creative Production 使用规范

Creative Production 负责品牌和视觉资产，不负责业务数据和法律文本。

使用范围：

- 产品视觉语言和风格路线探索。
- 业务流程说明图。
- 培训和操作引导插图。
- 财税资料包封面和目录。
- 董事长月报视觉。
- 发布汇报和演示素材。

工作流：

1. 基于真实页面、品牌约束和目标用户完成 style intake。
2. 生成 mood board 或视觉路线供选择。
3. 精确文字、数字、图表、Logo 和法律信息使用确定性 HTML、SVG 或文档模板。
4. ImageGen 只用于背景、质感、灯光、场景和非精确装饰层。
5. 最终资产保留来源、版本、尺寸、文件名和审查状态。

禁止：

- 用生成图片承载最终会计数字、税务数据或法律文本。
- 生成虚假产品界面、客户证明或合作方标识。
- 让视觉效果覆盖可读性、对比度和无障碍要求。

- AI 会计建议人工采纳率目标不低于 85%。
- 高风险误自动执行为 0。
- 核心页面 P95 不超过 2 秒。
- 普通 API P95 不超过 500ms。
- RPO 不超过 24 小时。
- RTO 不超过 4 小时。
- 所有高风险动作有最终授权和审计证据。
